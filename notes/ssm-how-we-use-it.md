# SSM (AWS Systems Manager) — What It Is and How We Use It

## What is SSM?

- **AWS Systems Manager** is a set of tools for managing EC2 and other AWS resources (patching, run command, parameter store, etc.).
- In this project we use two parts:
  - **Parameter Store** — key/value store for config and secrets (like a secure, central `.env`).
  - **SSM Run Command** — run a script on an EC2 instance from the pipeline without SSH.

---

## How we use SSM

### 1. Parameter Store — where we keep config and secrets

- We store configuration and secrets in **SSM Parameter Store** so they are not in code or in `.env` files.
- **Parameters we create (Terraform):**
  - **DATABASE_URL** — connection string for RDS (password included). Stored as **SecureString** (encrypted with KMS).
  - **COGNITO_REGION**, **COGNITO_USER_POOL_ID**, **COGNITO_APP_CLIENT_ID** — Cognito config for the backend. Stored as plain `String` (no secret).
  - **API_URL** — backend API URL (e.g. `http://api.coinbaby.click` or `http://<eip>:8000`). Used when building the frontend so the React app knows where to call the API.
- **Parameter names** follow a path, e.g. `/dalla/prod/DATABASE_URL`, `/dalla/prod/API_URL` (names are set via Terraform variables).

### 2. Who reads these parameters?

- **Backend (EC2):** When the pipeline deploys the backend, it sends a script to the EC2 instance via SSM Run Command. That script runs *on the EC2 instance* and calls `aws ssm get-parameter` to fetch DATABASE_URL and Cognito parameters, then starts the Docker container with those values as environment variables. So the backend never stores the DB password on disk; it gets it at deploy time from Parameter Store using the instance’s IAM role.
- **Frontend (CodeBuild):** The frontend build runs in CodeBuild. The buildspec runs `aws ssm get-parameter` for API_URL and Cognito parameters, exports them as `VITE_*` env vars, then runs `npm run build`. The built React app therefore has the correct API URL and Cognito settings baked in.

### 3. SSM Run Command — how we deploy the backend without SSH

- We do **not** deploy by SSHing into EC2. The pipeline uses **SSM Run Command** to run a script on the backend instance.
- **Flow:** CodeBuild runs `aws ssm send-command` with the document `AWS-RunShellScript` and the deploy script (pull image from ECR, get params from SSM, stop old container, start new container with env vars). The SSM agent on the EC2 instance receives the command and runs the script. CodeBuild then polls `ssm:GetCommandInvocation` until the command finishes and prints the output.
- **Requirements:** The EC2 instance has the **SSM agent** installed (we install it in `user_data` via `snap install amazon-ssm-agent`) and the instance role has **AmazonSSMManagedInstanceCore** so the agent can accept commands. The CodeBuild role has `ssm:SendCommand` (on the instance and the document) and `ssm:GetCommandInvocation` (to read status and output).

### 4. Why use Parameter Store instead of .env or hardcoding?

- **Secrets never in repo or on disk:** DATABASE_URL is generated by Terraform (`random_password`) and written once to Parameter Store as SecureString. No one needs to see or copy the password.
- **Single source of truth:** API URL and Cognito IDs are defined in Terraform; Parameter Store holds the values so both backend and frontend build can use the same config.
- **Access control:** Only roles that need them can read (EC2 backend role for backend params; CodeBuild role for the params it needs). Parameter Store supports IAM policies per parameter path.

### 5. Security choices

- **DATABASE_URL** is **SecureString** so it is encrypted at rest (KMS). Cognito IDs and API_URL are not secrets, so they are plain `String`.
- EC2 and CodeBuild use **IAM roles** to call `GetParameter`; no access keys stored anywhere.
- SSM Run Command uses the same IAM and agent model: no SSH keys or open port 22 required for deploy.

---

## Summary in one line

We use **SSM Parameter Store** to hold config and secrets (DB URL, Cognito, API URL) and **SSM Run Command** to run the backend deploy script on EC2 from the pipeline; nothing sensitive lives in code or in `.env` files.
